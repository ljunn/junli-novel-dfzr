# 返修与评审工作流

用于单章扩写、重写、润色、去 AI 味、问题定位和审稿。

## 一、先判断问题层级

先问：
- 问题是局部语言、单场景、章节结构，还是全书方向？
- 用户要的是评审意见、局部修订，还是直接出改稿？
- 这次允许改到哪一层：句子 / 段落 / 场景 / 整章 / 全书规则？

默认原则：
- 局部问题先局部修
- 只有主冲突、POV、章节回报或章末驱动力坏掉时，才升到整章重构

## 二、三档返修强度

### 轻修
适用：
- 少量套语
- 句群太匀
- 对白发虚但关系还在

动作：
- 把抽象词改成动作、物体、声音、温度、停顿
- 删除作者代说
- 打散连续同长度句群
- 把一部分台词改成动作或反应

### 中修
适用：
- 解释腔过重
- 对白承担太多设定说明
- 情绪只靠抽象词

动作：
- 重排“动作 -> 阻力 -> 反应 -> 后果”
- 给角色补潜台词、遮掩、试探、犹豫
- 把解释拆进环境反馈、关系压力和代价纹理

### 重修
适用：
- 章节悬浮
- 受压无补偿
- 结尾只有总结，没有追更驱动

动作：
- 重建“压力 -> 选择 -> 代价 -> 回报 / 新债”链路
- 重写结尾，让它停在动作、发现、危险或未决选择上
- 让人物关系或认知状态发生可感知变化

## 三、评审输出格式

默认先给 Findings，再决定是否改稿。

优先级：
- P0：主冲突、POV、世界规则、逻辑硬伤，必须先修
- P1：章节回报、节奏、对白、章末驱动力不足，建议先修
- P2：语言净化、表达收束、局部细节优化，可跟进

每条评审至少写：
- 问题位置
- 具体症状
- 为什么伤害阅读
- 最小修法

## 四、CLI 预检的边界

- `scripts/novel_pipeline.py review` 当前只是轻量预检，不是完整审稿器。
- CLI 现在只检查 4 类低成本规则：
  - 模板标题残留（如 `## 正文`）
  - 成串 AI 套语 / 解释腔
  - 章末总结腔
  - 正文过短、明显像壳子
- 爽点 / 毒点、POV 稳定性、世界规则、逻辑硬伤、章节回报密度，仍按上面的 P0 / P1 / P2 格式由模型或人工完整审。

## 五、四类高发病灶

1. 套话病
- “在这一刻”“心中涌起”“仿佛”“某种感觉”成串复用

2. 解释病
- 旁白替角色总结情绪
- 对白替作者讲设定

3. 均匀病
- 连续 4-6 句长度接近
- 句式总是“他/她 + 动词”

4. 同腔病
- 角色说话一个口气
- 人人都完整、理性、会总结

## 六、返修底线

- 不要为了去 AI 味把信息、爽点、钩子一起洗掉
- 不要把单点问题放大成世界观大修
- 不要越权改变用户没授权的角色关系或终局方向
- 不要先改稿再补理由，先判断问题层级再下手
